# 任务工作流-接口测试实施设计

> 本文档用于指导“任务工作流程页 + 接口测试与覆盖检查”功能的落地实现，偏开发设计与实施拆解。

---

## 一、实施目标

围绕任务清单中的单个任务，新增独立工作流程页，承载以下能力：

1. 展示并复用 TAPD 抓取后的需求 MD
2. 自动创建并维护开发执行 MD
3. 生成需求覆盖检查结果
4. 生成可执行接口测试计划
5. 执行接口测试并保留每次历史记录
6. 提供疑问项、阻塞项、失败摘要，供人工或外部 AI 修复后再次回归

---

- 触发 Playwright 页面辅助识别

### 6.2 执行阶段

建议统一阶段名称：

```text
loading_context

checking_coverage
generating_plan
preparing_data
running_cases
checking_db

```

### 6.3 `running_ui_trace` 阶段细分

建议拆分日志动作：

1. 登录系统
2. 打开目标页面
3. 执行动作模板
4. 采集网络请求
5. 过滤无关请求
6. 生成 `trace_json`

日志示例：

```text
页面识别 | 登录系统 | 成功 | env_id=3
页面识别 | 打开目标页面 | 成功 | /order/list
页面识别 | 执行动作模板 | 成功 | template=create_order
页面识别 | 采集请求样例 | 成功 | 共捕获 6 条请求
页面识别 | 过滤业务接口 | 成功 | 命中 2 条业务请求
```

### 6.4 日志写入规范

每个阶段至少写：

- 中文和英文阶段名注释
- 当前动作
- 执行结果
- 补充信息

示例：

```text
加载上下文 | 读取需求文档 | 成功 | fragment_id=req_xxx
覆盖检查 | 匹配接口 | 成功 | 命中 5 个接口
测试计划 | 生成用例 | 成功 | 共 12 条
接口测试 | 执行 case-003 | 失败 | code 断言不匹配
数据库校验 | 校验 orders 记录 | 成功 | 命中 1 条
```

## 七、覆盖分析实现设计

### 7.1 输入

- 需求 MD
- 开发执行 MD
- 当前分支 diff
- 接口定义摘要
- 路由 / 控制器摘要
- 数据库 schema 摘要（可选）

### 7.2 输出

结构固定为 `coverage_report.json`：

- `summary`
- `items`
- `questions`
- `blocked`

### 7.3 判断规则

- 每个 `covered / partial / missing` 必须带证据
- 需求不明确进入 `questions`
- 缺少造数能力进入 `blocked`
- 不允许仅凭模型猜测给出最终结论

## 八、测试计划实现设计

### 8.1 输入

- 覆盖分析结果
- 需求 MD
- 开发执行 MD
- 接口定义详情
- 环境信息
- 允许调用的前置业务接口列表
- 数据库 schema 摘要
- 可选的 `ui_api_trace.json`

补充说明：

- 默认基于代码、接口定义、数据库结构直接生成测试计划
- 只有在“需求偏页面化且接口映射不清”时，才额外引入 `ui_api_trace.json`

### 8.2 输出

结构固定为 `test_plan.json`，必须包含：

- `coverage_links`
- `preconditions`
- `api_cases`
- `open_questions`
- `blocked_items`

### 8.2.1 `test_plan.json` 字段校验建议

建议 Schema 级必填字段：

- `plan_name`
- `workflow_id`
- `source`
- `coverage_links`
- `preconditions`
- `api_cases`
- `open_questions`
- `blocked_items`

建议业务级校验规则：

- `api_cases` 至少 1 条
- `case_id` 全局唯一
- `preconditions.id` 全局唯一
- `requirement_id` 必须可映射到 `coverage_links`
- `api_id` 与 `api_uri` 至少存在一个
- `assertions` 至少 1 条

### 8.2.2 `api_cases` 结构建议

每条 `api_case` 推荐字段：

```json
{
  "case_id": "case-001",
  "name": "创建订单-正常流程",
  "requirement_id": "req-1",
  "api_id": 1001,
  "api_uri": "/api/order/create",
  "method": "POST",
  "priority": "P0",
  "tags": ["smoke", "order"],
  "request_data": {},
  "assertions": [],
  "db_checks": [],
  "depends_on": [],
  "timeout_ms": 5000
}
```

补充说明：

- `priority` 建议取值 `P0`、`P1`、`P2`
- `tags` 用于后续筛选和失败重跑
- `depends_on` 用于串联同批次中其他用例结果

### 8.3 硬约束

- 每条用例必须绑定需求点
- 前置数据优先走业务接口
- 不允许直接写数据库
- 数据库仅做只读校验
- 无法满足条件必须输出阻塞项

### 8.4 基于 Playwright 痕迹生成测试计划

当存在页面采集结果时，生成器需要执行以下步骤：

1. 读取 `trace_json` 中的请求链路
2. 过滤静态资源与无关请求
3. 将业务请求映射到接口定义中的 `api_id`
4. 从请求体中提取高价值参数样例
5. 从响应体中提取稳定断言字段
6. 判断请求属于前置造数还是正式验证接口
7. 转换为 `preconditions` 与 `api_cases`

这样可避免把浏览器动作脚本直接当成测试用例，而是沉淀为标准接口测试计划

请求过滤规则建议：

- 保留：`/api/` 下的业务请求、提交动作后的关键请求、与需求点强相关的查询请求
- 剔除：静态资源、埋点、心跳、配置下发、字典通用查询、与当前功能无关的自动刷新请求
- 标记为疑问：命中业务路径但无法映射到接口定义的请求

映射优先级建议：

1. 通过完整 URL 精确匹配接口定义
2. 通过路由规则匹配
3. 通过控制器方法或功能名模糊匹配
4. 仍无法匹配则进入 `open_questions`

## 九、测试执行实现设计

### 9.1 执行步骤

1. 读取最近一次测试计划，或在本次流程中生成计划
2. 按 `preconditions` 调用已有业务接口准备数据
3. 按 `api_cases` 调用 `/api/ApiRun`
4. 对断言进行逐条判定
5. 调用数据库只读校验能力验证落库结果
6. 生成 `test_report.json`
7. 写入 `tbl_task_test_run` 与 `tbl_task_test_case_result`

补充执行说明：

- `/api/ApiRun` 仍是第一期唯一正式执行器
- Playwright 采集结果只参与“识别接口”和“补充参数样例”，不直接承担批量测试执行
- 如果页面采集发现实际请求与接口定义不一致，应优先记录为疑问项并提示人工确认

变量解析建议：

- 支持 `{{pre-1.data.id}}` 这类前置结果引用
- 支持 `{{env.xxx}}` 读取环境变量
- 支持 `{{random.int}}`、`{{random.uuid}}` 这类轻量动态值
- 所有解析后的最终请求体都要保留快照

### 9.2 `/api/ApiRun` 对接约定

建议由测试执行器统一组装以下字段后调用：

```json
{
  "env_id": 3,
  "api_id": 1001,
  "api_uri": "/api/order/create",
  "method": "POST",
  "request_data": {
    "product_id": 123,
    "count": 2
  },
  "source": "task_workflow_test_execute",
  "source_id": 35
}
```

说明：

- `source` 标识调用来源，便于接口运行日志筛选
- `source_id` 可绑定当前 `test_run_id`
- 如 `api_id` 缺失，可临时退化为 `api_uri + method` 定位

### 9.3 测试执行错误码建议

建议统一错误码：

- `TASK_WORKFLOW_TEST_PLAN_NOT_FOUND`
- `TASK_WORKFLOW_PRECONDITION_FAILED`
- `TASK_WORKFLOW_VARIABLE_RESOLVE_FAILED`
- `TASK_WORKFLOW_APIRUN_FAILED`
- `TASK_WORKFLOW_ASSERT_FAILED`
- `TASK_WORKFLOW_DB_CHECK_FAILED`
- `TASK_WORKFLOW_REPORT_SAVE_FAILED`

建议错误返回结构：

```json
{
  "code": "TASK_WORKFLOW_VARIABLE_RESOLVE_FAILED",
  "message": "测试用例变量解析失败",
  "detail": {
    "case_id": "case-001",
    "expression": "{{pre-1.data.id}}"
  }
}
```

### 9.4 测试报告回显建议

前端展示测试报告时，建议每条结果保留：

- `case_id`
- `requirement_id`
- `api_id`
- `api_uri`
- `status`
- `failure_reason`
- `source_plan_version`

这样可以直接从结果跳转到：

- 用例定义
- 对应需求点
- 对应接口
- 对应计划版本

### 9.2 断言类型建议

- `status_code`
- `json_path`
- `json_type`
- `json_not_null`
- `json_contains`
- `response_time`
- `db_exists`
- `db_count`

### 9.3 错误处理原则

- 单条用例失败不应中断整个批次
- 阻塞项导致的跳过要明确标记为 `skipped`
- 结构化结果写入失败时，整个执行记为 `failed`

## 十、AI 编排 Agent 输入输出契约

### 10.1 覆盖分析 Prompt

输入：

- 需求 MD
- 开发执行 MD
- diff
- 接口定义摘要
- 路由 / 控制器摘要

输出：

- `coverage_report.json`

### 10.2 测试计划 Prompt

输入：

- 覆盖分析结果
- 需求 MD
- 开发执行 MD
- 接口详情
- 环境信息
- 可选的 `ui_api_trace.json`

输出：

- `test_plan.json`

额外要求：

- 若存在 `ui_api_trace.json`，优先使用其中的真实请求样例补充 `request_data`
- 若 `ui_api_trace.json` 与接口定义冲突，必须输出 `open_questions`
- 不允许把浏览器点击步骤直接输出为最终执行用例

建议附加生成规则：

- 输出前先判断每条候选请求属于 `prepare_api`、`target_api` 或 `noise`
- 至少为每个 `target_api` 生成 1 条主用例
- 如果存在稳定失败路径，可补 1 条异常用例
- 不要因为抓到了真实请求就省略数据库只读校验

### 10.3 页面辅助识别 Prompt

输入：

- 需求 MD
- 开发执行 MD
- 页面入口 URL
- 动作模板
- 环境信息

输出：

- `ui_api_trace.json`
- 请求过滤说明
- 接口映射建议

要求：

- 必须区分业务请求与噪音请求
- 必须为每条保留请求说明其用途
- 无法识别的请求进入 `open_questions`

### 10.4 失败总结 Prompt

输入：

- `test_report.json`
- 失败用例结果
- 覆盖分析结果

输出：

- 失败摘要
- 疑问项
- 可供人工或外部 AI 使用的问题定位说明

### 10.5 接口返回通用规范

建议以下接口统一返回：

- `code`
- `message`
- `data`

异步触发类接口的 `data` 建议至少包含：

- `async_task_id`
- `status`
- 可选的 `run_id` 或 `trace_id`

例如 `/api/task/workflow/ui-trace/generate`：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "async_task_id": 205,
    "trace_id": 88,
    "status": "pending"
  }
}
```

### 10.6 前端轮询或 SSE 建议

如果当前已有 SSE，优先复用 SSE 推送以下信息：

- 当前阶段变更
- 当前日志
- 当前进度
- 最终结果摘要

如果没有 SSE，建议前端按 2 到 3 秒轮询：

- `/api/task/workflow/test-run/info`
- `/api/task/workflow/ui-trace/info`

轮询停止条件：

- `status` 进入 `success`、`failed`、`cancelled`

## 十四、联调顺序建议

建议按以下顺序联调：

1. `create_or_get` + `info`，先打通页面初始化
2. `dev-plan/info` + `dev-plan/save`，打通文档编辑
3. `coverage/generate` + `coverage/info`，打通覆盖分析展示
4. `test-plan/generate` + `test-plan/info`，打通测试计划展示
5. `test-run/start` + `test-run/info` + `test-run/cases`，打通执行与详情
6. `db/check`，补齐数据库只读校验结果
7. `ui-template/*` + `ui-trace/*`，最后接入 Playwright 增强链路

这样可以保证：

- 核心 `API-only` 闭环最先可用
- Playwright 接入不会阻塞主流程交付
- 前后端可以分阶段验证页面状态与接口结构

## 十五、第一期验收标准

### 功能验收

- 任务清单能进入工作流程页
- 会自动生成开发执行 MD
- 能生成覆盖分析
- 能生成测试计划
- 能执行接口测试
- 能保留每次历史执行记录
- 能查看单次执行详情

### 质量验收

- 历史执行记录不覆盖
- 覆盖分析结果具备证据链
- 前置造数优先走已有业务接口
- 数据库工具无写权限
- 失败用例可查看请求 / 响应 / 断言 / 数据库校验详情

### 稳定性验收

- AI 输出结构不合法时能被拦截
- 单个用例失败不影响其他用例执行
- 页面刷新后可恢复当前任务状态
- SSE 或轮询可稳定获取执行进度

## 十七、结论

本实施设计的重点不是立刻做自动修复，而是把“工作流程页 + 覆盖检查 + 接口测试 + 历史记录”做成稳定底座。

当这套底座跑顺后，无论后续接入自动修复 Agent、扩展 UI 测试，还是沉淀测试平台能力，都会顺很多。
