# 首页命令待执行队列设计

## 目标

当首页命令已有任务执行中时，允许用户继续在输入框中输入完整命令并按回车，将该命令加入待执行队列。前一个任务完成后，系统自动按入队时保存的原始命令文本重新解析并串行执行。输入框右侧需要展示待执行命令列表，并支持删除单条待执行命令。

## 现状

- 首页命令入口位于 `web/src/components/Dashboard.vue`。
- 当前通过 `isExecuting` 控制执行态；执行中再次提交命令时，只会提示“正在执行其他命令，请稍候”。
- 输入框区域目前只有输入框、发送按钮和下一步提示，没有独立的待执行列表展示区域。
- 命令解析和执行逻辑已集中在 `parseInput`、`executeCommand`、`executeAction`、`finishExecution` 等方法中。

## 设计决策

### 1. 队列语义

- 待执行队列仅保存“用户按回车时的完整命令文本”。
- 后续执行时，按保存的原始命令文本重新走现有解析链路。
- 不保存命令栈快照、动态候选快照和执行上下文快照，避免与现有命令体系深度耦合。

### 2. 入队条件

- 只有当前输入已经满足执行条件时，才允许入队。
- 未补全完成的命令不入队，避免在队列中堆积无法执行的半成品命令。
- 脚本多步会话在交互步骤中先不开放额外排队，避免脚本会话状态与普通命令队列相互干扰。

### 3. 自动串行执行

- 当 `finishExecution` 被调用后，若待执行队列非空，则自动取出队首命令。
- 系统将该命令文本回填到输入链路并重新解析，再走统一执行逻辑。
- 若队首命令在重新解析时因环境变化导致无法执行，则按当前解析结果给出已有错误提示，并继续清空该条，不阻塞后续用户操作。

### 4. 页面展示

- 在输入框右侧增加待执行命令列表区域。
- 列表展示待执行数量、命令文本，并支持删除单条命令。
- 展示风格保持当前首页输入区视觉语言，不引入新的复杂交互。

## 数据结构

新增前端状态 `pendingCommandQueue`，每项包含：

- `id`: 前端唯一标识，用于列表渲染和删除。
- `rawCommand`: 入队时保存的完整命令文本。
- `createdAt`: 入队时间戳，用于保持稳定顺序和后续调试。

## 实现拆分

### 队列核心逻辑

将待执行队列的最小操作抽离为纯函数模块，至少覆盖：

- 创建队列项
- 入队
- 出队
- 删除指定队列项

这样可以在不依赖 Vue 运行时的情况下补回归测试。

### Dashboard 接入

- 新增响应式队列状态。
- 在 `executeCommand` 中区分“当前直接执行”和“执行中入队”。
- 在 `finishExecution` 中调度下一条命令。
- 在模板中增加右侧待执行列表展示。

## 风险与取舍

- 队列按原始文本重放，意味着动态目标可能在执行时已变化；这是本次明确接受的产品语义。
- 第一版不处理队列持久化，刷新页面后队列清空，避免引入额外存储与恢复逻辑。
- 第一版不支持拖拽排序，保持范围可控。

## 验证重点

- 执行中再次回车会入队，而不是直接报错。
- 待执行列表能展示多条命令。
- 删除某条待执行命令后，该命令不会被执行。
- 当前任务结束后，会自动串行执行下一条命令。
- 普通首页命令原有行为不被破坏。
